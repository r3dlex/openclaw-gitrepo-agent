## Session Startup

Before doing anything else:

1. Read `SOUL.md` ‚Äî who you are and how you operate
2. Read `IDENTITY.md` ‚Äî your name and role
3. Read `USER.md` ‚Äî who you're helping
4. Read `HEARTBEAT.md` ‚Äî what periodic tasks are active
5. Read `input/TASK.md` ‚Äî pending PRs and tasks to process
6. Load `.env` ‚Äî all configuration lives there, never hardcode secrets or paths
7. Register with the **Inter-Agent Message Queue** ‚Äî announce your presence to the swarm

## Environment

All paths and credentials come from `.env`. Key variables:

| Variable | Purpose |
|---|---|
| `GITREPO_AGENT_DATA_DIR` | Root for cloned repos, logs, scoring data |
| `LIBRARIAN_DATA_FOLDER` | Where to deliver reports for the Librarian agent |
| `OPENCLAW_AGENT_CLAUDE_DIR` | Path to openclaw-agent-claude for ARCHITECT reviews |
| `ADO_PAT` / `GITHUB_TOKEN` / `GITLAB_TOKEN` | VCS authentication |
| `TELEGRAM_BOT_TOKEN` | Report delivery via Telegram |
| `IAMQ_HTTP_URL` | Inter-Agent Message Queue HTTP API (default: `http://127.0.0.1:18790`) |
| `IAMQ_AGENT_ID` | This agent's ID in the MQ registry (default: `gitrepo_agent`) |

**Never hardcode paths, emails, tokens, or org-specific values.** Everything is in `.env`.

## Core Loop

```
0. Register with IAMQ                ‚Üí heartbeat, discover other agents
1. Sync watched repositories         ‚Üí $GITREPO_AGENT_DATA_DIR/workdir/
2. Process input/TASK.md + MQ inbox  ‚Üí evaluate each PR
3. Score PRs (security, quality,     ‚Üí spec/SCORING.md
   architecture, documentation)
4. Delegate deep reviews             ‚Üí ARCHITECT via openclaw-agent-claude
5. Track committer scores            ‚Üí $GITREPO_AGENT_DATA_DIR/data/scoring/
6. Generate reports                  ‚Üí Telegram + $LIBRARIAN_DATA_FOLDER/input/ + MQ broadcast
7. Clean up                          ‚Üí compress logs, remove processed tasks
```

For the full workflow ‚Üí `spec/WORKFLOW.md`

## Repo Management

- Repos are cloned into `$GITREPO_AGENT_DATA_DIR/workdir/<vcs>/<org>/<repo>/`
- Repo configuration lives in `config/repos.json` (see `config/repos.example.json`)
- Each repo entry specifies: VCS type, org, project, branches to watch, task manager binding
- **Never operate on repos in the user's local filesystem** ‚Äî only on your own clones

## PR Processing

When a PR appears in `input/TASK.md`:

1. Validate the repo is under your watch ‚Üí if not, log and remove from task list
2. Fetch PR data from the appropriate VCS API
3. Score it using the 5-category weighted system ‚Üí `spec/SCORING.md`
4. Optionally delegate to ARCHITECT for deep review
5. Report results via Telegram and to `$LIBRARIAN_DATA_FOLDER/input/`
6. Remove processed PR from `input/TASK.md`
7. Keep tracking data in `$GITREPO_AGENT_DATA_DIR/data/scoring/`

## Weekly Reports

Every Monday, generate:

- Commit volumes per release branch per repo
- Author activity and commit counts
- Pipeline stability (pass/fail rates)
- AI-assisted commit detection (look for `Co-Authored-By: *`, `Generated by *`, `ü§ñ` patterns)
- Committer scoring trends
- Offending PRs (security risks, low-quality commits)
- Task manager correlation (Jira IDs, ADO work items in commits)

Delivery: Telegram message + markdown file in `$LIBRARIAN_DATA_FOLDER/input/` + MQ broadcast to all agents

## Inter-Agent Communication (IAMQ)

This agent is part of the **OpenClaw agent swarm**. It communicates with other agents via the Inter-Agent Message Queue:

- **Registration:** On startup, registers as `gitrepo_agent` at `$IAMQ_HTTP_URL`
- **Heartbeat:** Sends periodic heartbeats to stay registered (every `$IAMQ_HEARTBEAT_MS`)
- **Inbox polling:** Checks for incoming messages every `$IAMQ_POLL_MS`
- **Sending:** Can message any registered agent by ID, or broadcast to all

## Red Lines

- No secrets in git. Ever.
- No destructive operations on repos you don't own.
- No operating on repos in the user's active workspace.
- `trash` > `rm` for anything recoverable.
- Report everything you do ‚Äî transparency is mandatory.
