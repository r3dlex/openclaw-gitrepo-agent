# PR and Committer Scoring System

The agent evaluates PRs using a 5-category weighted scoring system and maintains per-author rolling scores over time.

## Scoring Categories

| Category      | Weight | Focus Areas                                              |
|---------------|--------|----------------------------------------------------------|
| Security      | 25%    | Secrets detection, auth patterns, input validation, OWASP |
| Design        | 25%    | Architecture compliance, boundary respect, SOLID          |
| Style         | 15%    | Naming, formatting, consistency, clean code               |
| Practices     | 20%    | Testing, error handling, logging, DRY, edge cases         |
| Documentation | 15%    | Inline comments, PR description, migration notes          |

Each category is scored 0-100. The final score is the weighted average.

### Security (25%)

- Secrets and credential detection in code changes
- Authentication and authorization pattern correctness
- Input validation and sanitization
- OWASP Top 10 compliance
- Dependency vulnerability checks (known CVEs in added deps)

Critical security findings override the final verdict to `reject` regardless of overall score.

### Design (25%)

- Architecture compliance (does the change respect module boundaries?)
- Dependency direction (no upward dependencies in layered architecture)
- SOLID principle adherence
- Appropriate abstraction level
- Backward compatibility considerations

PRs scoring below 70% on Design trigger ARCHITECT delegation. See [WORKFLOW.md](WORKFLOW.md) Phase 3.

### Style (15%)

- Naming conventions (variables, functions, modules)
- Code formatting (language-specific linters)
- Consistency with existing codebase patterns
- Clean code principles (small functions, single responsibility)
- No magic numbers or unexplained constants

### Practices (20%)

- Test coverage for new/changed code
- Error handling (no swallowed exceptions, proper propagation)
- Logging (appropriate level and context)
- DRY compliance (no copy-paste duplication)
- Edge case handling (null checks, boundary conditions)

### Documentation (15%)

- Inline comments for non-obvious logic
- PR description quality (explains why, not just what)
- Migration notes when applicable
- API documentation for public interfaces
- ADR updates when architectural decisions change

## Verdict Mapping

| Score Range | Verdict                | Action                                    |
|-------------|------------------------|-------------------------------------------|
| 90-100%     | `approve`              | PR is good to merge                       |
| 70-89%      | `approve_with_comments`| Merge OK, but improvements suggested      |
| 50-69%      | `request_changes`      | Changes required before merge             |
| 0-49%       | `reject`               | Significant issues, needs rework          |

## Per-Author Rolling Score

Each author maintains a rolling score computed over a 1-year window:

- All PR scores within the window are included
- Recent PRs are weighted more heavily (exponential decay, half-life of 90 days)
- Minimum 3 PRs required before a rolling score is established
- Score stored in `$GITREPO_AGENT_DATA_DIR/data/scoring/<author-hash>.json`

## AI-Assisted Commit Detection

The agent flags commits that appear to be AI-assisted:

- `Co-Authored-By` headers referencing known AI tools (Copilot, Claude, ChatGPT, etc.)
- `Generated by` markers in commit messages or code comments
- Bot signatures in commit metadata
- Unusual commit patterns (large generated files, boilerplate-heavy changes)

AI-assisted commits are not penalized but are tracked separately in reports for transparency.

## Offending PRs

PRs with security findings are flagged separately regardless of overall score:

| Severity  | Action                                          |
|-----------|------------------------------------------------|
| Critical  | Immediate IAMQ urgent broadcast, verdict forced to reject |
| High      | Flagged in report, recommend blocking merge     |
| Medium    | Noted in report, included in weekly summary     |
| Low/Info  | Included in detailed report only                |

## Weekly Trends

Per-author weekly trend analysis:

- **Improving**: rolling score increased 5+ points over last 4 weeks
- **Stable**: score change within +/- 5 points
- **Declining**: rolling score decreased 5+ points over last 4 weeks

Trends are included in the Monday weekly report. See [COMMUNICATION.md](COMMUNICATION.md) for report format.

## Task Manager Correlation

When commits contain work item references (Jira IDs like `PROJ-123`, Azure DevOps IDs like `AB#456`), the agent:

1. Extracts the ID from commit messages and PR descriptions
2. Enriches the PR report with work item title and status
3. Includes task completion correlation in weekly reports (PRs linked to tasks vs. orphan PRs)

## Scoring Calibration

Weights and thresholds are configurable but should be adjusted cautiously. See [LEARNINGS.md](LEARNINGS.md) for calibration notes accumulated over time.
